官网技术方案对比

选择最适合 Omnigram 当前阶段的方案

A

纯静态站（HTML + Tailwind CSS）

单个 index.html + 少量 JS，GitHub Pages 部署

Pros

  • 零依赖，零构建，极简
  • AI 一次性生成整页，维护成本最低
  • 加载速度最快（无框架开销）
  • 适合当前阶段（只需一个 Landing Page）

Cons

  • 多页面时难以维护（无组件复用）
  • 后续加文档/博客需要迁移
  • 无 Markdown 支持，内容更新需改 HTML
B

Astro（静态站生成器）

组件化开发，支持 Markdown 内容，零 JS 输出

Pros

  • 组件复用，Landing Page + 文档 + 博客都能做
  • 默认零客户端 JS（性能接近纯静态）
  • Markdown/MDX 写文档，维护方便
  • Starlight 插件可一键生成文档站
  • 开源项目主流选择（Starlight 专为文档设计）

Cons

  • 需要 Node.js 构建环境
  • 学习成本（Astro 语法，但 AI 辅助下可忽略）
  • 比纯 HTML 多一层抽象
C

VitePress（文档站框架）

Vue 驱动的文档站，Landing Page + 文档一体

Pros

  • 文档体验极佳（搜索、侧边栏、导航）
  • Vue 组件嵌入 Markdown，灵活度高
  • 开箱即用的主题（类似 Vue/Vite 官网风格）
  • 构建快，热更新快

Cons

  • Landing Page 定制度受限于主题框架
  • 想要独特视觉风格需要大量覆盖默认样式
  • 偏「文档站」气质，不太像「产品官网」
D

Next.js SSG

React 生态，静态导出，功能最强

Pros

  • React 组件生态，UI 库丰富
  • 与 Server 端 Web UI（也是 React）技术栈统一
  • SSG 导出后性能好
  • AI 生成 React 代码质量最高

Cons

  • 对「官网」来说过于重量级
  • 客户端 JS bundle 较大
  • 构建配置复杂度高于 Astro
  • 杀鸡用牛刀

💡 推荐：B — Astro

理由：Omnigram 当前需要 Landing Page + 安装文档 + API 文档。Astro + Starlight 插件可以一套框架覆盖「品牌官网 + 技术文档」，且输出零 JS 的纯静态页面。竞品参考：Immich 官网用 Docusaurus，但 Astro Starlight 是更现代的替代。AI 辅助下 Astro 组件和 HTML 写法几乎一样简单。

## 三、部分实现（需补齐）

### 3.1 多格式元数据提取

- **已解决** ✅：已添加 FB2（XML 解析）、MOBI/AZW3（二进制头+EXTH 解析）完整元数据提取，PDF 封面提取（pdfcpu ExtractImages）
- 支持格式：EPUB / PDF / MOBI / AZW3 / FB2 — 均可提取标题、作者、描述、标签、封面等

### 3.2 Calibre 导入

- **已解决** ✅：CLI 逻辑重构为 `service/sys` 包的可复用函数，新增 `POST /sys/import/calibre` HTTP 端点（admin 权限）
- 封面数据现存入 BadgerDB（原 CLI 仅复制文件），CLI 与 API 共享同一实现

### 3.3 Web UI 功能深度

- **已解决** ✅：前端已扩展至 8 个页面/功能模块
  - 书架管理页面（创建/编辑/删除/浏览书架书籍）
  - 标签管理页面（标签列表/搜索/按标签浏览书籍/删除标签）
  - 统计仪表盘（总览卡片 + 每日阅读图表 + 阅读排行）
  - Calibre 导入 UI（集成到设置页，路径输入 + 结果展示）
  - 批量操作（图书馆多选模式，批量删除/批量添加标签）
  - 阅读注解展示（BookDetail 中显示高亮/笔记/书签）

## 五、需立即修复的问题

### 5.1 ~~OpenAPI 规范严重过时~~ ✅ 已解决

已采用 [swaggo/swag](https://github.com/swaggo/swag) 代码优先方案，从 Go 代码注解自动生成 API 文档：
- **生成文件**：`server/docs/`（docs.go, swagger.json, swagger.yaml）
- **Swagger UI**：服务运行后访问 `/swagger/index.html`
- **端点覆盖**：61 个路径，覆盖全部 ~70 个已实现端点
- **维护方式**：修改 handler 注解后执行 `cd server && make swagger` 即可同步
- **旧规范**：`docs/omnigram.openapi.spec.yaml` 归档保留，不再维护
- **编码规范**：已更新 `claude.md`，要求所有新增/修改的 handler 必须包含 swaggo 注解
